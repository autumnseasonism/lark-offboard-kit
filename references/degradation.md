# 降级与熔断策略

## 三档状态

每个模块的采集结果会被打上三种状态之一：

| 状态 | 图标 | 含义 |
|---|---|---|
| `ok` | ✅ | 完整采集，可直接信任 |
| `partial` | ⚠️ | 有截断或字段缺失，但已有数据可用；需标注触发原因 |
| `broken` | ❌ | 完全失败，走人工指引 |

## 决策树

```
采集调用
  │
  ├── rc=0 → 解析 JSON
  │         ├── 数据正常 → ok
  │         └── 达到分页上限/部分字段缺失 → partial
  │
  ├── rc≠0 且 stderr 含 "429"/"rate"/"too many"
  │         → 指数退避 1s→2s→4s 最多 3 次
  │         └── 仍失败 → 熔断该模块 → broken
  │
  ├── rc≠0 且 stderr 含 "scope"/"403"/"permission"
  │         → 标记 scope_missing=true → broken
  │         (报告里输出 self 模式重跑/管理员补 scope 两条指引)
  │
  ├── 单次失败 → breaker.failures++
  │         └── 连续 ≥3 次 → 熔断
  │
  └── 超时 (模块 30s / 全局 120s)
            → 未完成模块直接判 broken → 继续出报告
```

## 熔断器

- **作用域**：单 skill 运行内的内存状态，不跨运行持久化
- **阈值**：`MODULE_MAX_FAILURES = 3` 次连续失败 → 跳过该模块剩余重试
- **隔离**：模块之间完全独立，一个熔断不影响其他 4 个
- **报告透明度**：熔断原因写入 `modules[].note`，用户可据此追因

## 全局完整度告警

当 `broken` 模块数 `≥3`（五模块中过半失败），`summary.warning` 会写入：

> ⚠️ **≥3 模块失败，数据不完整，建议人工复核**

该信息会渲染到 Markdown 顶部，避免 HR 按残缺清单执行交接造成疏漏。

## 退出码语义

| 退出码 | 含义 |
|---|---|
| 0 | 全模块 ok |
| 1 | 前置失败（参数错/plan 找不到） |
| 2 | 部分模块降级（仍生成了报告，但不完整） |
| 3 | 认证/探测失败 |

CI/自动化可据此决定是否人工介入。

## 已知边界

- **群主模块 200 上限**：`im +chat-list` 分页默认上限 200，超过时标 `partial` 并提示当事人在飞书 App 内自助补齐。更彻底的方案是多页拼接，但会显著增加 API 调用量，权衡后保持当前行为。
- **审批的抄送人**：当前只查 `--assignee` 待处理实例；抄送/关注类型的审批未纳入，避免噪音。如需扩展可增补一个子查询。
- **日程周期判定**：仅靠 `recurrence` 字段非空判断；自然语言类固定会议（如"周会 1"但未设置 recurrence）会漏掉，建议接收人和当事人当面确认一次。
